RESTful API: แนวทางปฏิบัติและการออกแบบที่ดีที่สุดสำหรับนักพัฒนา

Intro

ปัจจุบันการพัฒนา API หรือ Microservice เพิ่มขึ้นอย่างมีนัยสำคัญ ซึ่งมีส่วนสำคัญในการใช้แลกเปลี่ยนข้อมูล กรณีมีแอปพลิเคชันในหลายรูปแบบ เช่น Web app หรือ Mobile app และ IoT โดยใช้ Restful API เป็นตัวกลางเชื่อมต่อระหว่างแต่ละอุปกรณ์ ซึ่งอนุญาตให้แลกเปลี่ยนข้อมูลระหว่าง Server กับ Client โดย Server รับ Requests จาก Client และส่ง Response ข้อมูลกลับ

API

คำว่า API นั้นย่อมาจาก Application Programming Interface ซึ่ง API จะเป็นตัวกลางที่อนุญาตให้แอปพลิเคชันแลกเปลี่ยนข้อมูลผ่าน Endpoint โดย Client จะใช้ API ส่งคำร้องขอ (Request) และตอบกลับข้อมูล (Response) (คล้าย ๆ การสื่อสารระหว่าง 2 อุปกรณ์โดยมี API เป็นตัวกลาง)

Note: Endpoints คืออะไร?

Endpoint คือ Term ของ URL Path เป็นเหมือนภาษาพูดที่ใช้เรียก Path ของ API โดยจะเรียก URLs ทั้งหมดว่า API แต่ละ URL จะเรียกว่า Endpoint

  • Public endpoints: Endpoint ที่สามารถเข้าถึงได้แบบสาธารณะ ไม่จำเป็นต้องมีการยืนยันตัวตนก่อนใช้งาน
  • Authenticated endpoints: Endpoint ที่จำเป็นต้องมีการยืนยันตัวตนก่อนใช้งาน

REST

คำว่า REST นั้นย่อมาจาก Representational State Transfer ซึ่ง Rest คือคำเรียกในเชิงแบบสถาปัตยกรรมที่ใช้ในการพัฒนาเว็บเซอร์วิส โดยนำเสนอครั้งแรกโดย Roy Fielding

RESTful API

RESTful นั้นพัฒนามาจากรากฐานสถาปัตยกรรมรูปแบบของ REST และใช้ HTTP Methods เรียกว่า RESTful web services โดยทั่วไป API นั้นช่วยให้การแลกเปลี่ยนข้อมูลนั้นง่ายขึ้นมาก ๆ โดยจะอนุญาตให้นักพัฒนา Integrated ฟังก์ชันบางอย่างจาก API ที่เป็น Third-party Services เพื่อสร้างฟังก์ชันอะไรบางอย่างของตัวเอง อย่างกรณีเช่น Uber และ Grab ใช้ Google Map API สำหรับการทำระบบนำทาง (Navigation) โดยช่วยให้ลดเวลาพัฒนาลงอย่างมากในการพัฒนาระบบเองจากศูนย์

RESTful API Design

API นั้นสามารถสร้างได้จากหลากหลายภาษา (Server side script Language Programming) ไม่ว่าจะเป็น PHP Ruby JS Java Python และ Golang เป็นต้น

คำศัพท์

  • Collections Resource เช่น /api/v1/users สำหรับหลาย items
  • Instance Resource เช่น /api/v1/users/123 สำหรับ item เดียว

โครงสร้าง API Endpoint อ้างอิงตามด้านล่างนี้ โดย URL โครงสร้างควรเป็นไปตามแบบลำดับชั้น และสามารถคาดการณ์ได้เพื่อให้ทำความเข้าใจและการใช้งานง่าย

คำอธิบาย

Scheme เป็นตัวที่ระบุ Protocol ซึ่งโดยปกติจะเป็น http หรือ https (มี s คือ secure หรือมีระบบความปลอดภัยกว่า)

Base หรือ ชื่อโดเมน หรือ DNS หรือบางคนเรียก ชื่อโฮสต์ (Host Name) เป็นองค์ประกอบสำคัญของ URL ที่ช่วยให้ผู้ใช้สามารถเข้าถึงเว็บไซต์ที่ต้องการได้อย่างถูกต้อง ชื่อโดเมนของแต่ละเว็บไซต์จะแตกต่างกันไปตามที่ผู้พัฒนาเว็บกำหนดแต่ส่วนท้ายของชื่อจะมีการกำกับตามประเภทของเว็บไซต์นั้น ๆ

Version หรือ API Versions คือส่วนที่บอกว่า APIs มีรุ่นและการเปลี่ยนแปลงอย่างไร และปัจจุบันเป็นรุ่นอะไร — API Versions สำคัญอย่างไรนั้น คำตอบนั้นคือ เมื่อทำการเผยแพร่ API เป็น Public แล้ว ตัวAPI จะต้องไม่เกิดการ Breaking (อยู่ดีๆ ใช้งานไม่ได้) ของ API โดยไม่แจ้งผู้ใช้ให้ทราบล่วงหน้า ดังนั้น API Versioning จะช่วยให้ป้องกันการเกิด Request ที่ไม่มีอยู่

Path คือเส้นทางที่นำไปสู่ไฟล์หรือตำแหน่งต่างๆ ของข้อมูลบนเว็บไซต์ โดย Path จะอยู่ด้านหลังของชื่อโดเมนเสมอและคั่นเครื่องด้วยเครื่องหมายทับ (/)

Query Parameter หรือ Query String คือ ค่าหรือตัวแปรเพิ่มเติมเพื่อการเข้าถึงข้อมูลที่ลึกหรือเฉพาะมากขึ้นบนเว็บไซต์ เปรียบเสมือนเป็นการกรองข้อมูลให้แยกย่อยลงไปอีกตามที่ผู้ใช้ต้องการ มักปรากฎเครื่องหมาย “?” ตามด้วยค่าพารามิเตอร์เช่นสีหรือขนาดของสินค้าบางอย่าง ซึ่งหากมีพารามิเตอร์มากกว่า 1 ค่า URL จะแสดงค่าเป็นคู่โดยใช้เครื่องหมาย “&” คั่นระหว่างค่า และแบ่งคู่ด้วยเครื่องหมาย “=”

รูปแบบของ Resource Names ที่ดี

- ควรเป็นคำนาม อย่าใช้คำกริยาในการอธิบาย

- ชื่อ Resource ควรเป็นเอกพจน์สำหรับแบบ Instance Resource

- ชื่อ Resource ควรเป็นพหูพจน์สำหรับแบบ Collections Resource

- ชื่อ Resource ควรตั้งเป็นตัวพิมพ์เล็ก (Lower-case) และเครื่องหมาย - (Hyphen) สำหรับคั่นคำ

รูปแบบของชื่อ Query Parameters ที่ดี

- ตัวอักษรใน Query string ควรใช้ _ (Underscore) สำหรับคั่นคำ

- Query parameters ควรเป็นตัวพิมพ์เล็กทั้งหมด (Lower-case)

- Query parameters ใช้เท่าที่จำเป็น เช่นการ Filters และ Sorting

- Query parameters พึงระวังไว้เสมอว่าไม่ควรใช้ตัวอักษรที่เป็นข้อมูล Sensitive

รูปแบบของชื่อ Fields หรือ Attribute Names ที่ดี

- รูปแบบของชื่อควรสอดคล้องกับรูปแบบของข้อมูล JSON

- ควรเป็นตัวพิมพ์เล็กทั้งหมด (Lower-case) ใช้ _ (Underscore) สำหรับคั่นคำ

- ไม่ควรใช้ Prefix อย่างเช่น is_ หรือ has_ สำหรับ keys ประเภทแบบบูลีน

- Fields ที่เป็นรูปแบบ Arrays ควรใช้ชื่อที่เป็นคำนามพหูพจน์ ตัวอย่าง

  • authenticators-contains หรือ authenticators
  • products-contains หรือ products

การเลือกใช้ HTTP Method ให้เหมาะสมกับการใช้งาน

ปกติการสร้าง API เราจะไม่ใช้ verb ของ API นั้นๆให้มาอยู่ใน path เช่น /GetUser, /CreateUser การทำเเบบนี้ถือว่าผิดหลักการในการออหเเบบ ทุกคนก็อาจจะสงสัยว่า "เเล้วเราจะรู้ได้อย่างไรว่า API ตัวไหนมี Action อะไร ทำอะไรได้บ้าง" เลยได้มีการกำหนด HTTP Method ขึ้นมา โดย HTTP Method ที่สำคัญมัดังนี้

1. POST เป็น method สำหรับสร้างข้อมูลใหม่ โดยจะส่งข้อมูลในการสร้างผ่านทาง body เช่นแทนที่จะเขียนว่า /createUsers ก็เขียนเป็น /users ที่มี method POST แทน

2. PUT เป็น method สำหรับแก้ไขข้อมูลที่มีอยู่แล้วในระบบ (ทั้ง object)หรือถ้าไม่มีในระบบจะสร้างข้อมูลใหม่

3. PATCH เป็น method สำหรับแก้ไขข้อมูลที่มีอยู่แล้วในระบบ คล้ายกับ PUT แต่ PATCH ใช้กรณีอัพเดทแค่ไม่กี่ค่า

4. GET เป็น method สำหรับเรียกข้อมูล และจะไม่มีผลกระทบอะไรกับข้อมูลในระบบ

5. DELETE เป็น method สำหรับลบข้อมูลที่มีอยู่ในระบบ

Note: HTTP Method

HTTP Method เป็นตัวสื่อถึง action ของ API นั้นๆ โดยที่เราไม่จำเป็นต้องใส่ action จริงๆ เข้าไปใน path

HTTP methods

HTTP methods คือ กริยา หรือบางทีเรียก HTTP Verbs โดยเป็นการดำเนินการบนของมาตรฐาน HTTP Request ดังนี้

HTTO Method Description
GET สำหรับการร้องขอ Resource
POST สำหรับเพิ่ม Resource ใหม่ หรือ สำหรับสั่งเพื่อการดำเนินการบางอย่าง เช่น POST ส่งข้อความ
PUT สำหรับแก้ไขข้อมูลใน Resource (ควรใช้เฉพาะกรณีต้องการ Replace ทุก Fields เท่านั้น)

DELETE สำหรับลบข้อมูลใน Resource

HTTP response status code

คำอธิบายเเละความหมายของ HTTP/1.1:Statatus Code Definitiond

  • 1xx informational response - server เเจ้งให้รู้ว่าคำร้องขอสามารถดำเนินงานต่อได้

Filtering, Sorting, Pagination

Filtering

Filtering คือความสามารคในการ Filter เเละ Sort ของ Collections ใน API เพื่อที่จะอนุญาตให้ควบคุมการเเสดงผลข้อมูลใน API ให้มีความยืดหยุ่น

  • การกำหนดจำนวนการเเสดงผลด้วยวิธีต่างๆ (เเต่วิธีนี้ไม่เเนะนำ) โดยเป็นส่วนหนึ่งของ API URL
  • Filter paramenters ไม่เป็นส่วนหนึ่งของการกำหนด Resource ให้ใช้ Query parameters แทนเพื่อ Filter และ Sort ตามความต้องการ

ข้างล่างนี้จะเป็นความแตกต่างของเทคนิคที่ใช้กับ Filter และ Sort:

Output Selection

ผู้ใช้ API สามารถระบุ Attributes ที่เขาต้องการสำหรับการตอบกลับในส่วนของ Response payload ด้วยการระบุ Attributes ใน Query parameters

ตัวอย่างการตอบกลับสำหรับ first_name และ last_name เท่านั้น

?attributes=first_name,last_name

Simple Filtering

Attributes สามารถใช้ Filter กับ collection ของ resources

ตัวอย่างการ Filter ชื่อ

?last_name=Johnson

จะ Filter ออกจาก Collection ของ resources ด้วย attribute last_name ที่เท่ากับ Johnson

ใช้สัญลักษณ์เท่ากับ = สำหรับเทคนิคที่ใช้เปรียบเทียบข้อมูลที่เท่ากัน และสำหรับการดำเนินการอื่น ๆ ดูในหัวข้อ Advanced Filtering

ตัวอย่างการ Filter วันเกิด

?last_name=Johnson&date_of_birth=1999-12-31

จะ Filter ข้อมูลออกจาก Collection ของ resources ด้วย attribute last_name ที่เท่ากับ Johnson และ date_of_birth เท่ากับ 1999-12-31

Advaned Filtering

มีบางสถานการณ์ที่การ Filter อย่างง่ายนั้นไม่ตรงกับความต้องการและจำเป็นต้องมีแนวทางอื่นที่ครอบคลุมมากขึ้น อย่างเช่นใช้ตัวกรองสัญลักษณ์ดังต่อไปนี้เพื่อกำหนดตรรกะการกรองที่ซับซ้อนมากขึ้น

สามารถกำหนดแอตทริบิวต์หลายรายการที่มีตัวดำเนินการและเงื่อนไขต่างกันได้ในพารามิเตอร์ตัวกรองการค้นหา

รองรับตัวดำเนินการต่อไปนี้ :

1. >= มากกว่าหรือเท่ากับ

2. > มากกว่า

3. < น้อยกว่า

4. <= น้อยกว่าหรือเท่ากับ

5. = เท่ากับ

6. != ไม่เท่ากับ

หรือไม่ว่าจะเป็นเงื่อนไข AND , OR ก็กำหนดได้

ตัวอย่างการใช้ AND , OR :

?filters=creation_date >= 2001-09-20T13:00:00 and creation_date <= 2001-09-21T13:00:00 and first_name like 'Fred' and post_code=3000

จะแสดง collection ของ resources เมื่อ creation_date อยู่ระหว่าง 2001-09-20 1pm และ 2001-09-21 1pm และ first-nameคือ Fred และ post_code คือ 3000

Match Case Sensitivity

  • ตามคำแนะนำทั่วไปการ Return ข้อมูลที่คล้ายกันจะดีกว่าการไม่ส่งคืนรายการที่ไม่ตรงกันเลย
  • ค่าที่กำหนดของการกรองควรพิจารณาจะไม่คำนึงถึงขนาดตัวพิมพ์
  • ไม่ว่าคุณจจะไม่คำนึงถึงขนาดตัวพิมพ์ แต่ก็ควรจัดทำเป็นเอกสารไว้อย่างชัดเจน

Sorting

การที่ข้อมูลสามารถกำหนดตามลำดับการเรียงที่เฉพาะเจาะจงได้มักเป็นความต้องการจากแอปพลิเคชันไคลเอนต์ ดังนั้นจึงเป็นเรื่องสำคัญที่จะต้องให้ความยืดหยุ่นแก่ผู้ใช้ในการดึงข้อมูลตามลำดับที่พวกเขาต้องการ

พารามิเตอร์สองตัวที่ใช้ในการเรียงลำดับมีดังนี้

Query Parameter Description
sort ในการจัดเรียงข้อมูลเช่น asc หรือ desc
sort_fields พารามิเตอร์ที่ใช้จัดเรียง id หรือ name

sort_fields เป็นพหูพจน์เนื่องจากมีทั้งตัวเลือกต่อไปนี้

  • ?sort=asc&sort_fields=name,last_modified
  • ?sort=desc&sort_fields=name&sort_fields=last_modified

คำค้นหาทั้งสองนี้ควรเรียงลำดับตามชื่อจากนั้นตามวันที่เเก้ไขล่าสุด

Pagination

ระบบเลขหน้าำตามคำเเนะนำดังนี้

1. ส่งคำขอในเวลาที่เหมาะสม (เช่น request < 2 วินาที)

2. ตรวจสอบให้เเน่ใจว่าปริมาณข้อมูลที่ตอบกลับยังคงอยู่ใส่วนที่จัดการได้ (เช่น 500kb)

3. ตรวจสอบปริมาณข้อมูลในการตอบกลับสามารถจัดการได้ง่ายเพื่อรองรับประสบการณ์ของผู้ใช่ที่ดี (UX)

Parameters

เมื่อใช้การเเบ่งหน้าให้ใช้พารามิเตอร์ต่อไปนี้ :

Query Parameter Description Example
page หน้าเพจที่ผู้ใช้ต้องการ page=1 (กำหนเค่าเริ่มต้น 1)
limit จำนวนผลลัพธ์ต่อหน้าที่ผู้ใช่ต้องการ limit=10 (กำหนเค่าเริ่มต้น 10)

ตัวอย่างโครงสร้าง URL ดังนี้ :

Page 1: /customers?page=1&limit=10
Page 2: /customers?page=2&limit=10
    ...
Page 50: /customers?page=50&limit=10

โครงสร้างนี้เข้าใจง่ายเนื่องจากมีความเรียบง่าย และสามารถรวมไว้ใน UI เช่น ลิงก์ต่าง ๆ ได้โดยตรง

Common Terms

พารามิเตอร์ต่อไปนี้เป็นพารามิเตอร์ทั่วไปใน REST APIs แต่ห้ามใช้ในข้อกำหนดดังนี้

offset และ limit การใช้สองสิ่งนี้เป็นเรื่องปกติในฐานข้อมูล SQL และเป็นตัวเลือกที่ดีเมื่อคุณต้องการให้ผลลัพธ์ออกมาอย่างต้องการ อย่างไรก็ตามการใช้งานพารามิเตอร์ offset อาจเป็นเรื่องยากดังนั้นจึงแนะนำให้ใช้ page แทน

since และ limit ดึงข้อมูลทุกอย่าง "ตั้งแต่" โดยสิ่งเหล่านี้มีประโยชน์เมื่อต้องการให้ไคลเอนต์สามารถ "ซิงค์" กับข้อมูลได้อย่างมีประสิทธิภาพ อย่างไรก็ตามโดยทั่วไปแล้วต้องมีการกำหนดการเรียงลำดับเริ่มต้นไว้ให้ชุดผลลัพธ์เพื่อให้มีความสำคัญอย่างมีนัย

Metadata

เมื่อมีการแบ่งหน้าให้กำหนดสิ่งนี้ไว้ แนะนำอย่างยิ่ง สำหรับ API เพื่อที่จะให้คำแนะนำผู้ใช้ในเนื้อหาการตอบกลับ (response body) ว่าพวกเขาอยู่ในหน้าใด ถ้าหากไม่กำหนดไว้ผู้ใช้จะต้องกำหนดให้ตรวจสอบคำขอซึ่งมักจะต้องมีโค้ดอย่างไม่จำเป็นเพื่อตรวจสอบพารามิเตอร์การสืบค้น

API Versioning

API Versions คือส่วนที่บอกว่า APIs มีรุ่นและการเปลี่ยนแปลงอย่างไร และปัจจุบันเป็นรุ่นอะไร

ทำไมต้องมี API Versions

ทำไมต้องมี API Versions เพราะเมื่อคุณเผยแพร่ API เป็น Public แล้วดังนั้น API ของคุณจะต้องไม่เกิดการ Breaking (อยู่ดี ๆ ใช้งานไม่ได้) ของ API โดยไม่แจ้งผู้ใช้ให้ทราบล่วงหน้า ดังนั้น API Versioning จะช่วยให้ป้องกันการเกิด Request ที่ไม่มีอยู่ เช่น จากการปรับปรุง Endpoints ดังตัวอย่างนี้

https://api.abc.com/User

//Response result
[
    {
        "title": "...",
        "text": "..."
    }
        ...
]

ถ้าหากวันนึงเราเเก้ไข API นี้ใหม่

https://api.abc.com/User
                    
//Response result
{
    "data" [
        {
            "title": "...",
            "text": "..."
        }
            ...
    ]
}

จะเห็นว่า API https://api.abc.com/User ส่วนปรับปรุงใหม่มีส่วนที่แตกต่างจากเดิมซึ่งจะทำให้เกิดปัญหากับ Clients ที่ใช้ API นี้อยู่โดยถ้าไม่ได้รับการปรับปรุงตาม API ใหม่ ดังนั้นเมื่อนำ API Versions มาช่วยแก้ปัญหาจะได้ดังนี้

#API Version ก่อนการแก้ไขสามารถใช้กับ Clients ได้ไม่มีปัญหา https://api.abc.com/v1/User

#API Version หลังการแก้ไข และรอการปรับปรุง Clients เพื่อความเข้ากันได้ https://api.abc.com/v2/User

คำเเนะนำการออกเเบบ API Versions

โดยที่ :

  • โดย API version ควรจะ กำหนดเป็นส่วนหนึ่งใน URI
  • โดย API version ควรจะ กำหนดไว้อยู่เสมอ (เช่น ค่าเริ่มเป็นเวอร์ชันล่าสุด)
  • โดย API version ควรจะ ระบุด้วย 'v'
  • ถ้าผู้ใช้งานพยายามเข้าถึง API ด้วยการไม่ระบุเวอร์ชัน ดังนั้นเขาก็ควรได้รับข้อความผิดพลาดตอบกลับ 404
  • ถ้าผู้ใช้งานพยายามเข้าถึง API ด้วยการไม่ระบุ resource (เช่น GET .../api/v1) การตอบกลับอาจจะประกอบด้วยข้อมูลพื้นฐานเกี่ยวกับ API (เช่น ชื่อ API, และรายละเอียดอื่น ๆ)

โดยอาจจะมี API Version หลายอันได้ ตัวอย่าง :

  • .../api/v1
  • .../api/v2

Metadata

เพื่อให้การสนับสนุน API ของคุณ ควร ให้มีการตอบกลับคำขอ GET ไปยัง Base URI ของ API และส่งคืนรูปแบบข้อมูล Metadata ต่อไปนี้การตอบกลับ :

  • api_name: ชื่อ API
  • api_version: API Version กับ Major และ minor versions
  • api_released: วันที่ล่าสุดที่ปรับปรุง API
  • api_documentation: ลิงก์สำหรับ API Documentation
  • api_status: สถานะที่บ่งบอกว่า API นั้น Active หรือ Deprecated

สามารถเพิ่มข้อมูล Metadata อื่นๆ สำหรับการตอบกลับ (ถ้าต้องการ)

ตัวอย่าง :

//GET /namespace/v1

//HTTP 200 OK
                    
{
    "api_name": "namespace",
    "api_version": "1.1.3"
    "api_released": "2021-04-06"
    "api_documentation": "https://api.example.com/namespace/v1/docs"
    "api_status": "active"
}

คำเเนะนำอื่นๆ

แนวทางและหลักการบางประการในการตัดสินใจว่า เมื่อใดจึงจำเป็นต้องใช้ API เวอร์ชันใหม่ :

  • คุณไม่ควรสร้างเวอร์ชัน API ใหม่ เมื่อคุณเพิ่มฟิลด์ / แอตทริบิวต์ให้กับ Resources ที่มีอยู่ ถ้าหากผู้ใช้สามารถเพิกเฉยต่อสิ่งเหล่านั้นได้ (กล่าวคือหากผู้ใช้ต้องไม่เพิ่มสิ่งเหล่านั้นในคำขอ)
  • คุณไม่ควรสร้างเวอร์ชัน API ใหม่ ถ้าหากคุณเพิ่ม Resources ใหม่ โดยที่ลูกค้าไม่จำเป็นต้องใช้มันโดยตรง
  • คุณต้องสร้างเวอร์ชัน API ใหม่ ถ้าหากคุณแก้ไขทรัพยากรที่มีอยู่ในลักษณะที่จะส่งผลกระทบต่อไคลเอนต์ (เช่นฟิลด์บังคับใหม่การลบฟิลด์ ... )
  • คุณต้องสร้างเวอร์ชัน API ใหม่ ถ้าหากคุณมี Resources ใหม่ โดยหากผู้ใช้ต้องได้ใช้มันด้วย
  • คุณต้องสร้างเวอร์ชัน API ใหม่ ถ้าหากคุณสร้างประเภทข้อผิดพลาดใหม่

เอกสาร (API Docs) เป็นสิ่งที่สำคัญสำหรับนักพัฒนาในการใช้ API โดย API ที่แตกต่างกันจะมีลักษณะการทำงานที่แตกต่างกัน ซึ่งต้องใช้พารามิเตอร์ที่แตกต่างกัน เช่น วิธีการ HTTPS การ Response data ของ API ดังนั้นนักพัฒนาส่วนใหญ่จะชอบเอกสารที่ดีสำหรับใช้อ้างอิงการใช้งาน API

มีเครื่องมือดี ๆ มากมาย เพื่อที่จะช่วยนักพัฒนาในการสร้างเอกสาร API ดังนี้ :

Swagger : ออกแบบและสร้างโมเดล API ตามมาตรฐานตามข้อกำหนด เพิ่มประสบการณ์ (UX) ของนักพัฒนาด้วยเอกสาร API แบบโต้ตอบได้

Slate : Slate ช่วยให้คุณสร้างเอกสาร API ที่สวยงาม ฉลาด ความสามารถการตอบสนอง ที่สำคัญ Slate คือโครงการโอเพ่นซอร์สและใช้งานได้ฟรี

Security Features

เป็นที่นิยมในการใช้คุณสมบัตินโยบายความปลอดภัยที่มีอยู่ในแพลตฟอร์ม API Gateway แทนที่จะใช้นโยบาย (Policy) การเข้าถึงใน back-end API ตรง ๆ แต่วิธีใดก็ตามควรจะมีรูปแบบความปลอดภัยดังนี้

ความปลอดถัย เหตุการณ์ของ API รายละเอียด ความจำเป็นในการนำไปใช้
HTTP verbs Listeners->Path HTTP verbs ต่าง ๆ สามารถเลือกใช้ได้จาก Path ของ API ที่กำหนดไว้ ควร
SSL Listeners ใช้ SSL protocol (เช่น TLS 1.2) ควร
Digital Certificate Filter -> Integrity สามารถ Filters สำหรับใช้ Signature เพื่อการตรวจสอบตัวอย่างเช่น API Token หรือ JWT sign เป็น Optional หรือขึ้นอยู่กับ Business requirements
JWT Filter -> Integrity ข้อความสามารถเข้ารหัสให้อยู่ในรูปแบบ JWT เป็น Optional หรือขึ้นอยู่กับ Business requirements
API Keys Filter -> Authentication สามารถ Filters สำหรับการ Authentication ผู้ใช้งาน เช่น API Keys หรือ API Token ควร
OAuth Filter -> OAuth สามารถใช้ OAuth สำหรับมอบสิทธิ์เข้าถึง Data resources สำหรับผู้ใช้งาน เป็น Optional หรือขึ้นอยู่กับ Business requirements
CORS Listeners->Path CORS สามารถจำกัดการเข้าถึงในระดับ Path level ควร

ประเด็นอื่น ๆ เกี่ยวกับความปลอดภัย

ก่อนการพัฒนา API นักพัฒนาต้องระบุช่องโหว่และการแก้ไขข้อบกพร่องด้านความปลอดภัยเป็นไปได้ที่อาจเกิดขึ้น (Threat Modeling) มิฉะนั้นอาจจะโดนคุกคามฐานข้อมูลและระบบ

  • ใช้ SSL เสมอสำหรับทุก APIs
  • ใช้มาตรฐานสำหรับการ Authentication และ Authorization เช่น JWT หรือ OAuth2 สำหรับการ Authenticate API ก่อนการ Responding ข้อมูลใด ๆ ที่สำคัญในขั้นตอน Request
  • อย่าเก็บข้อมูลที่สำคัญ (Sensitive data) ใน JWT payload ที่ง่ายสำหรับการ Decoding
  • ใช้การ Encryption ทั้งหมดสำหรับข้อมูลที่สำคัญ (Sensitive data) อย่าเก็บ Raw password ใน databases โดยให้ Encrypt Password ก่อนการบันทึกเสมอ
  • ใช้วิธีการกำหนด Rate Limiting สำหรับการโจมตีแบบ DDoS attacks/brute-force
    • Return 429 “Too Many Requests” ใช้สำหรับการแจ้งเกิดเหตุการณ์ Too many requests ที่มาจาก Origin เดียวกันอย่างรวดเร็ว
    • เพิกถอน (Revoke) ข้อมูล Client credential หรือทำ Blacklist ได้เสมอหากไคลเอ็นต์ละเมิดการใช้งาน API
  • Validate inputs request ทั้งหมดก่อน Responding คำขอใด ๆ เพราะการปล่อยให้ข้อมูลที่ไม่ถูกต้องนี้เข้าสู่แอปพลิเคชันอาจทำให้เกิดผลลัพธ์ที่ไม่ดีที่ไม่สามารถคาดเดาได้
  • อย่าส่งผ่านข้อมูลที่ละเอียดอ่อน (Sensitive data) ใน URL เช่น รหัสผ่าน JWT token หรือ API keys เนื่องจากข้อมูลนี้อาจจะเก็บไว้ในบันทึกของเบราว์เซอร์และเซิร์ฟเวอร์
    • ตัวอย่างที่ไม่ควร https://api.example.com/namespace/v1/auth/login/username=abcd123456789&password=123321
    • URL ด้านบนแสดงรหัสผ่าน ดังนั้นอย่าใช้รูปแบบนี้ในระบบ Authentication

Characteristics of Good Restful APIs

  • พิจารณาขนาดความยาวของ Parameter lists ด้วย
  • สามารถใช้ Pagination และ Sorting ด้วย date กำหนดตัวเลขขนาดของ Records ต่อหน้าได้
  • กำหนด Versioning ของ API อย่างเหมาะสม
  • อ่านง่ายและใช้งานง่ายสำหรับ ควรทำตามที่ชื่อและโปรโตคอลที่แนะนำ
  • เพื่อประสิทธิภาพพยามให้การทำงานไม่มีวิธีใดขึ้นอยู่กับผลลัพธ์ของอีกวิธีหนึ่ง (Stateless)
  • สามารถจัดการข้อผิดพลาดได้ด้วย HTTP Status codes
  • ความเหมาะสมของ Parameter names ความเหมาะสมของ Naming conventions ความเหมาะสมของ Lowercase letters เพื่อใช้ใน URI paths (อย่าใช้คำย่อ)
  • ความพร้อมใช้งานและปลอดภัยสูง
  • มีความสามารถของ API Versioning Pagination และ Sorting
  • มีเอกสาร API Docs ที่ดีเพราะ Developer ส่วนใหญ่นั้นชอบเอกสารอธิบายที่ดี
  • ออกแบบหรือทำให้เรียบง่ายเข้าไว้

How to scale the API

  • สามารถกำหนด Rate Limiting ได้
  • สถาปัตยกรรม Microservice ซึ่งแต่ละบริการได้รับการออกแบบมาเพื่อให้ทำงานแยกฟังก์ชันของการทำงานออกเป็นบริการส่วน ๆ ซึ่งหากฟังก์ชันใดเกิดมีปัญหาหรือใช้งานไม่ได้ ระบบอื่น ๆ ก็จะยังสามารถทำงานต่อได้ ดั้งนั้นควรใช้บริการหลายตัวบนอินสแตนซ์ที่ทำงานบนเครื่องหลายเครื่อง ด้วยด้านหลังตัวโหลดบาลานเซอร์
  • การทำ Caching static files
  • ใช้ CDN (Content Delivery Network)
  • ทำ indexing กับบางข้อมูล
  • ทำ Filtering และ Pagination แทนที่จะแสดงข้อมูลทั้งหมด
  • หากใช้ RDB ให้ทำ Normalized database และใช้ SQL joins ให้น้อยที่สุด
  • นำ API Gateway มาช่วยในการจัดการ Microservice เช่น Kong

Conclusion

ประเด็นทั้งหมดนี้เป็นเพียงความเห็นส่วนตัว โดยสิ่งเหล่านี้ไม่ใช่กฎที่ตายตัว แต่เป็นเพียงเคล็ดลับจากประสบการณ์ส่วนตัว   เพราะเนื่องจากการออกแบบและการกำหนดรูปแบบมาตรฐาน API นั้นไม่ได้มีมาตรฐานตายตัว โดยสิ่งที่สำคัญควรอธิบายและตกลงกับทีมพัฒนาให้ชัดเจนนั้นเป็นสิ่งสำคัญ ไม่ว่าจะเป็นการตั้งชื่อ URL Path การตั้งชื่อ Attributes การกำหนด JSON format เป็นต้น

แต่สิ่งที่สำคัญสูงสุดเมื่อ API ได้ Public แล้ว สำหรับความปลอดภัยในบาง Endpoints นั้นก็เป็นเรื่องที่สำคัญ การ Revoke ต้องสามารถทำได้หากไคลเอ็นต์ละเมิดการใช้งาน API และพิจารณาการใช้งาน API แบบ Stateless และ Stateful ควบคู่กันได้ในบางกรณี

นอกจากนี้ยังมีข้อมูลที่น่าสนใจสามารถเอามาปรับใช้ร่วมกับการออกแบบและใช้อ้างอิงเพื่อนำมาปรับใช้เกี่ยวกับการออกแบบสเปกของ JSON รายละเอียดดังนี้ JSON API Specification

Authentication REST API: การตรวจสอบสิทธิ์ REST API

Authentication REST API คืออะไร

เป็นกระบวนการตรวจสอบข้อมูลประจำตัวของผู้ใช้หรือแอปพลิเคชันก่อนที่จะอนุญาตให้เข้าถึงทรัพยากรของ REST API

การตรวจสอบสิทธิ์ REST API ทำงานแตกต่างจากการตรวจสอบสิทธิ์บนเว็บแบบเดิมๆ ซึ่งโดยทั่วไปเราอาจจะต้องเข้าสู่ระบบด้วยชื่อผู้ใช้และรหัสผ่าน จากนั้นจึงสร้างเซสชันสำหรับชื่อผู้ใช้ในหน้าเข้าสู่ระบบของเว็บไซต์ การตรวจสอบสิทธิ์ REST จะไม่เก็บข้อมูลเซสชันไว้ตามค่าเริ่มต้น คำขอแต่ละรายการจะดำเนินการโดยอิสระ โดยตรวจสอบตัวเองโดยไม่ต้องพึ่งพาการโต้ตอบในอดีต

ความเเตกต่างระหว่าง การตรวจสอบสิทธิ์ กับ การอนุญาต

การแยกแยะระหว่างการพิสูจน์ตัวตนและการอนุญาตก็มีความสำคัญเช่นกัน การพิสูจน์ตัวตนจะยืนยันว่าคุณเป็นใคร ในขณะที่การอนุญาตจะกำหนดว่าคุณทำอะไรได้บ้าง

ตัวอย่างเช่น การตรวจสอบสิทธิ์ API อาจตรวจสอบข้อมูลประจำตัวของแอปพลิเคชันของคุณ แต่การอนุญาตจะกำหนดว่าแอปพลิเคชันของคุณมีสิทธิ์ในการแก้ไขข้อมูลหรือเข้าถึงทรัพยากรที่ละเอียดอ่อนหรือไม่ แม้ว่าจะผ่านการตรวจสอบสิทธิ์แล้ว ผู้ใช้ก็อาจไม่สามารถเข้าถึงทรัพยากรหรือฟังก์ชันทั้งหมดของ API ได้

การตรวจสอบสิทธิ์ REST API ถือเป็นแนวทางการรักษาความปลอดภัยชั้นแรกที่ดีสำหรับ REST API จากนั้นจึงสามารถจับคู่กับการอนุญาตเพื่อให้แน่ใจว่าโครงสร้างความปลอดภัย เช่น หลักการสิทธิ์ขั้นต่ำ (PoLP) จะถูกนำไปใช้ใน API ของคุณอย่างถูกต้อง

วิธีการตรวจสอบสิทธิ์ REST API ทั่วไป

การตรวจสอบสิทธิ์ (Authentication) ใน REST API เป็นขั้นตอนสำคัญที่ช่วยให้ API มีความปลอดภัย โดยมีวิธีการต่างๆ มากมายสำหรับการนำการตรวจสอบสิทธิ์ไปใช้ใน REST API เพื่อป้องกันการเข้าถึงที่ไม่ได้รับอนุญาตและปกป้องข้อมูลสำคัญของผู้ใช้งาน การเลือกใช้วิธีการตรวจสอบสิทธิ์ที่เหมาะสมขึ้นอยู่กับข้อกำหนดและข้อจำกัดของระบบ API รวมถึงความสะดวกในการใช้งานสำหรับผู้พัฒนาและผู้ใช้งาน ต่อไปนี้เป็นวิธีการตรวจสอบสิทธิ์ REST API ทั่วไป

การยืนยันตัวตนพื้นฐาน

การตรวจสอบสิทธิ์พื้นฐานเป็นวิธีการตรวจสอบสิทธิ์ HTTP ที่ง่ายและรวดเร็ว หากต้องการเข้าถึงจุดสิ้นสุด API ผู้ใช้จะต้องส่งชื่อผู้ใช้และรหัสผ่านไปยังผู้ให้บริการ API ในส่วนหัวการตรวจสอบสิทธิ์ของคำขอ ผู้ให้บริการ API จะตรวจสอบข้อมูลประจำตัวและในกรณีที่สำเร็จ ผู้ให้บริการจะให้สิทธิ์การเข้าถึงแก่ผู้ใช้

มาดูกันว่าคุณจะนำการตรวจสอบสิทธิ์ขั้นพื้นฐานไปใช้ใน Python ได้อย่างไร ตัวอย่างนี้แสดงให้เห็นว่าคุณสามารถเข้ารหัสข้อมูลประจำตัวด้วยไลบรารี base64 ได้อย่างไร จากนั้นจึงเพิ่มข้อมูลเหล่านี้ลงในส่วนหัว Authorization ด้วยคำนำหน้า Basic จากนั้นคุณสามารถใช้ส่วนหัวนี้เพื่อเข้าถึงจุดสิ้นสุด API ด้วยวิธี requests.get() ได้:

import requests, base64
url = 'https://restapi.test.com/api/get_endpoint'
credentials = 'user:password'
encoded = base64.b64encode(credentials.encode('ascii'))
headers = {
  'Authorization': f'Basic {encoded}'
}
response = requests.get(url, headers=headers)
print(response.status_code)
print(response.json())

การตรวจสอบสิทธิ์ขั้นพื้นฐานนั้นง่ายต่อการนำไปใช้งานและมีค่าใช้จ่ายในการประมวลผลบนเซิร์ฟเวอร์ต่ำมาก อย่างไรก็ตาม ข้อมูลประจำตัวที่ส่งเป็นส่วนหนึ่งของคำขอดังกล่าวจะเดินทางโดยไม่ได้เข้ารหัส (อาจมีการเข้ารหัสเท่านั้น) ซึ่งทำให้ข้อมูลเหล่านี้เสี่ยงต่อการถูกดักฟัง นอกจากนี้ เนื่องจากการตรวจสอบสิทธิ์ขั้นพื้นฐานนั้นค่อนข้างเรียบง่ายและตรงไปตรงมา คุณจึงขาดการควบคุมขั้นสูง เช่น การจัดเตรียมและการเพิกถอนสิทธิ์การเข้าถึงแบบละเอียด

การใช้การตรวจสอบสิทธิ์ขั้นพื้นฐานนั้นสมเหตุสมผลเมื่อสร้าง API ภายใน เพื่อวัตถุประสงค์ในการทดสอบ หรือเมื่อจัดการกับไคลเอนต์ที่เชื่อถือได้ในสภาพแวดล้อมที่ปลอดภัย อย่างไรก็ตาม หากคุณกำลังจัดการกับข้อมูลที่ละเอียดอ่อนใน API ที่เผยแพร่สู่สาธารณะ หรือกรณีการใช้งานของคุณต้องการการควบคุมการเข้าถึงแบบละเอียด คุณควรพิจารณาทางเลือกที่ปลอดภัยยิ่งขึ้นบางส่วนในรายการนี้

การยืนยันตัวตนด้วยรหัส API

ในการยืนยันตัวตนด้วยรหัส API ผู้ให้บริการ API จะกำหนดรหัสเฉพาะให้กับไคลเอนต์แต่ละรายที่เข้าถึง API ไคลเอนต์ต้องรวมรหัส API ของตนเป็นส่วนหนึ่งของคำขอเพื่อยืนยันตัวตน คีย์ API สามารถรวมไว้ในที่ใดก็ได้ในคำขอ เช่น ส่วนหัว เนื้อหา หรือพารามิเตอร์การค้นหา โดยท้ายที่สุดแล้วจะขึ้นอยู่กับการออกแบบ API และจะแจ้งให้นักพัฒนาทราบผ่าน เอกสาร ประกอบAPI

ตัวอย่างต่อไปนี้แสดงวิธีการตรวจสอบสิทธิ์ด้วย REST API ที่ใช้คีย์ API ใน Python โดยใช้ไลบรารีคำขอ โดยจะเพิ่มสตริงที่มีคีย์ API ในส่วนหัวการอนุญาตพร้อมคำนำหน้า api-key และใช้ส่วนหัวนี้เมื่อส่งคำขอไปยัง API

import requests
url = 'https://restapi.test.com/api/get_endpoint'
api_key='some_secret_key'
headers = {
  'Authorization': f'api-key {api_key}'
}
response = requests.get(url, headers=headers)
print(response.status_code)
print(response.json())

การตรวจสอบสิทธิ์ด้วยคีย์ API นั้นทำได้ง่ายมาก คุณสามารถกำหนดชุดสิทธิ์เฉพาะให้กับคีย์ API แต่ละตัวได้ ทำให้เป็นตัวเลือกยอดนิยมสำหรับสถานการณ์ที่ระดับความปลอดภัยปานกลางก็เพียงพอแล้ว ข้อดีอีกประการหนึ่งก็คือช่วยให้คุณสามารถเพิกถอนคีย์ API ได้ หากคีย์ถูกบุกรุก เซิร์ฟเวอร์สามารถเพิกถอนคีย์นั้นได้ เพื่อให้ไคลเอนต์สามารถขอคีย์ใหม่ได้

อย่างไรก็ตาม วิธีนี้ปลอดภัยในระดับปานกลางเท่านั้น หากเครือข่าย API มีความเสี่ยง ผู้โจมตีสามารถดักจับคีย์ API และนำไปใช้ในทางที่ผิดได้จนกว่าจะหมดอายุหรือถูกปิดใช้งานด้วยตนเอง ดังนั้น หลีกเลี่ยงการใช้วิธีนี้เมื่อต้องจัดการข้อมูลที่มีความละเอียดอ่อนสูง หรือเมื่อคุณต้องติดตามกิจกรรมของผู้ใช้เพื่อวัตถุประสงค์ในการบันทึกหรือตรวจสอบ

การยืนยันตัวตนแบบโทเค็น

การตรวจสอบสิทธิ์แบบใช้โทเค็น—เรียกอีกอย่างว่าการตรวจสอบสิทธิ์ผู้ถือ—เป็นวิธีการตรวจสอบสิทธิ์ที่ได้รับความนิยมซึ่งใช้โทเค็นการเข้าถึงเพื่อยืนยันตัวตนของผู้ใช้

ในขณะที่คีย์ API ใช้เพื่อระบุไคลเอ็นต์ เช่น ไซต์หรือแอปที่โทรออก โทเค็นจะถูกใช้เพื่อระบุผู้ใช้เนื่องจากมีข้อมูลเพิ่มเติมเกี่ยวกับผู้ใช้ โทเค็นนั้นแตกต่างจากวิธีการอื่นๆ ที่กล่าวถึงไปแล้ว ซึ่งหมายความว่าคุณสามารถสร้างโทเค็นได้ทันทีโดยใช้โฟลว์การสร้างโทเค็นเฉพาะที่สร้างขึ้นใน API วิธีนี้ช่วยให้สามารถสร้างโทเค็นเฉพาะที่มีอายุสั้นได้ตามกรณีการใช้งานทันทีของผู้ใช้

โทเค็นสามารถปรับแต่งเพื่อให้เฉพาะกับอุปกรณ์หรือผู้ใช้ ซึ่งหมายความว่าคุณยังสามารถ (ในระดับหนึ่ง) ตรวจสอบความถูกต้องของโทเค็นบนเซิร์ฟเวอร์ของคุณก่อนดำเนินการตามคำขอ ซึ่งจะช่วยเพิ่มความปลอดภัยให้กับ API ของคุณอีกด้วย

มาดูตัวอย่างการตรวจสอบสิทธิ์ด้วย REST API ที่ต้องใช้โทเค็นผู้ถือ หากต้องการเข้าถึงจุดสิ้นสุดของ API เพียงเพิ่มโทเค็นที่มีคำนำหน้าว่า "Bearer" ลงในส่วนหัวการอนุญาตของคำขอ:

import requests
url = 'https://restapi.test.com/api/get_endpoint'
token='fhu78ej3-fh37-fy67-56ed-56ddgc5dte45'
headers = {
  'Authorization': f'Bearer {token}'
}
response = requests.get(url, headers=headers)
print(response.status_code)
print(response.json())

การตรวจสอบสิทธิ์โดยใช้โทเค็นเป็นหนึ่งในวิธีการตรวจสอบสิทธิ์ REST API ที่ปลอดภัยที่สุด โดยจะช่วยลดความเสี่ยงในการถูกดักจับได้ เนื่องจากสามารถเข้ารหัสได้และมักมีระยะเวลาสั้น นอกจากนี้ ยังสามารถควบคุมการเข้าถึงได้อย่างละเอียดอีกด้วย

อย่างไรก็ตาม โทเค็นจำเป็นต้องมีการออกแบบและพิจารณาโครงสร้างพื้นฐานอย่างรอบคอบ โดยทั่วไปแล้ว โทเค็นจะมีความซับซ้อนในการใช้งานมากกว่าวิธีอื่น และกระบวนการสร้างโทเค็นอาจเพิ่มภาระในการประมวลผลให้กับเซิร์ฟเวอร์

การใช้การตรวจสอบสิทธิ์แบบโทเค็นนั้นสมเหตุสมผลที่สุดเมื่อต้องจัดการข้อมูลที่ละเอียดอ่อนใน API สาธารณะหรือเมื่อสร้างระบบนิเวศ API ที่ซับซ้อนและปรับขนาดได้ หากความเรียบง่ายและประสิทธิภาพการใช้ทรัพยากรเป็นข้อกังวลหลักของคุณ คุณควรใช้วิธีการที่ง่ายกว่านี้

คู่มือสำหรับการทดสอบ API ด้วย Postman

แนะนำคุณเกี่ยวกับการทดสอบ API ด้วย Postman เราจะครอบคลุมการเขียนกรณีทดสอบด้วยตนเองเพื่อใช้เทคนิคอัตโนมัติด้วย Postman มาเริ่มต้นด้วยการอธิบายว่า API คืออะไรและทุกสิ่งที่คุณจำเป็นต้องรู้เกี่ยวกับ API ต่อไป เราจะแนะนำ Postman คืออะไรและประโยชน์ของ Postman ในการทดสอบ API นอกจากนี้ เราจะแนะนำคุณเกี่ยวกับคำแนะนำทีละขั้นตอนในการใช้ Postman สำหรับการทดสอบ API

API คืออะไร และเหตุใดการทดสอบจึงสำคัญ?

API เป็นตัวย่อของ Application Programming Interfaces API ใช้เพื่ออำนวยความสะดวกในการบูรณาการระหว่างระบบต่างๆ กล่าวโดยง่าย API ทำหน้าที่เป็นสะพานเชื่อมการสื่อสารที่ช่วยโต้ตอบกับส่วนประกอบซอฟต์แวร์และแอปพลิเคชันต่างๆ นอกจากนี้ API ควรปฏิบัติตามโปรโตคอลและมาตรฐานบางอย่าง ซึ่งช่วยให้ผู้พัฒนา API สามารถเข้าถึงฟังก์ชันการทำงานหรือข้อมูลเฉพาะจากบริการภายนอก ไลบรารี หรือระบบต่างๆ ได้ API มีบทบาทสำคัญในการพัฒนาซอฟต์แวร์สมัยใหม่ ช่วยให้สร้างระบบนิเวศที่ซับซ้อนและเชื่อมต่อถึงกันได้

API ทำงานอย่างไร?

คำขอ API ถูกสร้างขึ้นโดยแอปพลิเคชันไคลเอนต์ และมีการเรียก API ไปยังแอปพลิเคชันปลายทางเพื่อประมวลผลคำขอ API การเรียกนี้มีประเภทคำขอและประกอบด้วยจุดสิ้นสุด ส่วนหัว และบางครั้งอาจมีเนื้อหาคำขอด้วย เมื่อเซิร์ฟเวอร์ API ได้รับคำขอแล้ว เซิร์ฟเวอร์จะประมวลผลคำขอและส่งการตอบกลับไปยังแอปพลิเคชันไคลเอนต์ตามข้อมูลที่ร้องขอ

API มีโปรโตคอลและมาตรฐานประเภทใดบ้าง?

  • API ที่ปฏิบัติตามข้อจำกัดจากสถาปัตยกรรม REST จะอยู่ภายใต้โปรโตคอล REST มาตรฐาน REST ประกอบด้วยวิธีการ HTTP เช่น GET, POST, PUT, PATCH และ DELETE
  • SOAP เป็นโปรโตคอลที่ใช้สำหรับแลกเปลี่ยนข้อมูลที่มีโครงสร้างในการใช้งานเว็บเซอร์วิส โดยจะกำหนดชุดกฎสำหรับการจัดรูปแบบข้อความและการสื่อสาร

เหตุใดการทดสอบ API จึงมีความสำคัญ?

การทดสอบ API มีความสำคัญเนื่องจากช่วยให้มั่นใจได้ว่าการสื่อสารข้อมูลระหว่างระบบซอฟต์แวร์มีความถูกต้อง นอกจากนี้ยังช่วยป้องกันความไม่สอดคล้องหรือการตีความข้อมูลผิด นักพัฒนา API กำลังพัฒนาฟีเจอร์ใหม่ การปรับปรุง และการแก้ไขข้อบกพร่อง ดังนั้น API จึงมีการเปลี่ยนแปลงบ่อยครั้ง การทดสอบ API ไม่เพียงแต่ช่วยให้แน่ใจถึงฟังก์ชันใหม่เท่านั้น แต่ยังช่วยให้แน่ใจด้วยว่าการเปลี่ยนแปลงใหม่นั้นไม่ส่งผลกระทบต่อฟังก์ชันที่มีอยู่ นอกจากนี้ การทดสอบอย่างละเอียดยังช่วยระบุคอขวดด้านประสิทธิภาพที่อาจเกิดขึ้นและช่องโหว่ด้านความปลอดภัยที่อาจส่งผลกระทบต่อระบบโดยรวม

ประเภทการทดสอบ API มีอะไรบ้าง?

การทดสอบยูนิตจะตรวจสอบความแม่นยำของจุดสิ้นสุดแต่ละจุดในการจัดการคำขอตั้งแต่พารามิเตอร์เสริมไปจนถึงการตอบสนองต่อข้อผิดพลาด

การทดสอบการบูรณาการจะตรวจสอบการโต้ตอบระหว่างส่วนประกอบซอฟต์แวร์ที่แตกต่างกันเพื่อให้แน่ใจว่ามีการสื่อสารระหว่างกัน

การทดสอบแบบครบวงจรจะตรวจสอบการเดินทางของผู้ใช้ในจุดสิ้นสุดหลายจุด ตรวจจับปัญหาในสถานการณ์ที่ซับซ้อนและแบบครบวงจรก่อนที่ผู้ใช้จะพบเจอการทดสอบประสิทธิภาพจะประเมินการตอบสนองและการใช้ทรัพยากรภายใต้เวิร์กโหลดระดับต่างๆ เพื่อปรับประสบการณ์และประสิทธิภาพของผู้ใช้ให้เหมาะสมที่สุด

การทดสอบโหลดช่วยยืนยันความน่าเชื่อถือของ API ระหว่างช่วงที่มีปริมาณการรับส่งข้อมูลสูงสุด โดยจำลองปริมาณคำขอสูง และวัดเวลาตอบสนองและข้อผิดพลาด

การทดสอบสัญญาณจะทำให้แน่ใจถึงการทำงานของ API ผ่านการแสดงที่มนุษย์และเครื่องสามารถอ่านได้ สร้างรากฐานสำหรับข้อตกลงระดับบริการและป้องกันการละเมิดสัญญาในการเผยแพร่ใหม่ๆ

Postman คืออะไร?

Postman เป็นเครื่องมือทดสอบ API ที่มีผู้ใช้มากกว่า 20 ล้านคน เครื่องมือนี้ได้รับความนิยมอย่างล้นหลามในอุตสาหกรรมไอทีในหมู่ผู้พัฒนาและผู้ทดสอบ อินเทอร์เฟซที่เรียบง่ายและเป็นมิตรต่อผู้ใช้ช่วยได้เมื่อต้องจัดทำเอกสาร ออกแบบ และทดสอบ API คุณสามารถสร้างคำขอไปยังจุดสิ้นสุดของ API ส่งข้อมูลประเภทต่างๆ และตรวจสอบการตอบกลับได้อย่างง่ายดายด้วย Postman อินเทอร์เฟซผู้ใช้แบบกราฟิกที่เรียบง่ายและให้ข้อมูลทำให้แม้แต่ผู้เริ่มต้นที่มีประสบการณ์น้อยกว่าก็สามารถโต้ตอบกับ API ได้อย่างง่ายดายโดยไม่ต้องเจาะลึกโค้ดที่ซับซ้อน

แนวทางปฏิบัติที่ดีที่สุดที่คุณสามารถปฏิบัติตามได้เมื่อคุณทดสอบ API ด้วย Postman คืออะไร

สิ่งพื้นฐานที่สุดคือการมีสภาพแวดล้อมการทดสอบเฉพาะของคุณเอง คุณสามารถสร้างสภาพแวดล้อมการทดสอบแยกต่างหากด้วยข้อมูลจำลองสำหรับการทดสอบที่ปลอดภัยและค้นหาข้อจำกัดของแอปพลิเคชันของคุณ ไม่เพียงแต่กรณีการทดสอบที่ประสบความสำเร็จเท่านั้น แต่คุณยังสามารถตรวจสอบกรณีการทดสอบที่ผิดพลาดหรือผิดพลาดได้ด้วยอินพุตที่แตกต่างกัน นอกจากนี้ คุณยังสามารถทำการทดสอบ API อัตโนมัติเพื่อให้แน่ใจว่าครอบคลุมอย่างสม่ำเสมอและลดข้อผิดพลาดของมนุษย์ให้เหลือน้อยที่สุด โดยบูรณาการเข้ากับกระบวนการ CI/CD เมื่อทำได้ นอกจากนี้ คุณยังสามารถดำเนินการทดสอบ API ในทุกขั้นตอนของการพัฒนา ตั้งแต่การออกแบบไปจนถึงการผลิต เพื่อระบุและแก้ไขปัญหาในระยะเริ่มต้นและรักษาคุณภาพของ API

ความท้าทายในการทดสอบ API มีอะไรบ้าง?

การทดสอบ API มีปัญหาในการรับรองความครอบคลุมและการออกแบบการทดสอบที่มีประสิทธิภาพ โครงสร้าง API ที่เรียบง่ายทำให้การสร้างกรณีทดสอบง่ายขึ้น ในขณะที่ API ที่ซับซ้อนซึ่งมีพารามิเตอร์จำนวนมากจำเป็นต้องมีการออกแบบที่ซับซ้อน เพื่อรักษาความเกี่ยวข้อง ให้ปรับปรุงการครอบคลุมการทดสอบตามความต้องการทางธุรกิจที่เปลี่ยนแปลงไป การจัดลำดับการเรียกใช้ API อย่างเหมาะสมและการสำรวจเงื่อนไขขอบเขตและประสิทธิภาพมีความสำคัญต่อการทดสอบที่ประสบความสำเร็จ

การทดสอบสถานการณ์เชิงลบอาจมีความซับซ้อนและใช้เวลานาน โดยมักต้องมีการสร้างทรัพยากรใหม่เพื่อจำลองกรณีการใช้งานเฉพาะ นอกจากนี้ จำนวน API และกรณีการใช้งานที่เพิ่มมากขึ้นยังทำให้ต้องจัดสรรทรัพยากรบุคคลจำนวนมากสำหรับการทดสอบด้วยตนเอง ซึ่งอาจทำให้เกิดความล่าช้าในการปรับใช้ การรับรองประสิทธิภาพและฟังก์ชันการทำงานที่สม่ำเสมอต้องอาศัยการทดสอบ API ในสภาพแวดล้อมที่แตกต่างกันหลายแบบ ตั้งแต่การพัฒนาไปจนถึงการผลิต ซึ่งเป็นความท้าทายที่สำคัญเช่นกัน เนื่องจากเป็นงานที่ค่อนข้างซับซ้อน

การสร้างข้อมูลการทดสอบ โดยเฉพาะอย่างยิ่งสำหรับการทดสอบประสิทธิภาพนั้นต้องใช้ความพยายามและใช้เวลานาน ซึ่งส่งผลต่อประสิทธิภาพการทดสอบโดยรวม การจำลอง API: การจำลองพฤติกรรมของ API ผ่านการล้อเลียนนั้นมีความจำเป็น โดยเฉพาะอย่างยิ่งเมื่อ API ที่แน่นอนกำลังอยู่ในระหว่างการพัฒนา ซึ่งจะเพิ่มความซับซ้อนในการทดสอบอีกชั้นหนึ่ง ดังนั้น จึงชัดเจนว่าการรักษาสมดุลระหว่างความเรียบง่ายและความซับซ้อนนี้จะช่วยเพิ่มความแม่นยำในการทดสอบ API

Postman มีประโยชน์ต่อการทดสอบ API มากเพียงใด?

  • การรับรองคุณภาพ: การทดสอบ API รักษาความไว้วางใจและชื่อเสียงโดยการรับรองประสิทธิภาพที่เชื่อถือได้
  • การตรวจจับปัญหาในระยะเริ่มต้น: การทดสอบ API แบบ Shift-left ค้นหาและแก้ไขข้อบกพร่องได้อย่างทันท่วงที
  • การอนุรักษ์ทรัพยากร: การทดสอบ API อัตโนมัติช่วยประหยัดเวลาและส่งเสริมนวัตกรรม
  • การวนซ้ำอย่างรวดเร็ว: การทดสอบ API ในไปป์ไลน์ CI/CD ช่วยให้สามารถเผยแพร่โดยไม่มีข้อบกพร่องได้บ่อยครั้ง
  • การจัดระเบียบการทดสอบที่มีประสิทธิภาพ: อินเทอร์เฟซของ Postman รองรับการสร้างคอลเลกชันที่สามารถนำกลับมาใช้ซ้ำได้ ช่วยปรับปรุงการจัดระเบียบและการแบ่งปันการทดสอบ
  • การบูรณาการ CI ที่ราบรื่น: Postman บูรณาการกับเครื่องมือ CI ชั้นนำได้อย่างราบรื่น ช่วยให้มองเห็นการสร้าง API ควบคู่ไปกับการทดสอบและการพัฒนา
  • เอกสารที่อ่านได้โดยเครื่อง: เอกสารที่อ่านได้โดยเครื่องของ Postman ช่วยส่งเสริมการสื่อสารที่ชัดเจนและความเข้าใจเกี่ยวกับ API สำหรับสมาชิกในทีมและผู้มีส่วนได้ส่วนเสีย
  • การทำงานร่วมกันแบบเรียลไทม์: ความสามารถในการควบคุมเวอร์ชันและการทำงานร่วมกันของ Postman ส่งเสริมการแชร์และการทำงานเป็นทีมแบบเรียลไทม์ ซึ่งช่วยลดระยะเวลาของรอบการพัฒนา
  • การออกแบบและการล้อเลียน: คุณลักษณะการออกแบบและการล้อเลียนของ Postman ช่วยลดความจำเป็นในการตั้งค่าเซิร์ฟเวอร์แบ็คเอนด์เบื้องต้น ช่วยเร่งการพัฒนา API

จะใช้ Postman เพื่อทดสอบ API ได้อย่างไร?

มาเริ่มกระบวนการทีละขั้นตอนนี้โดยดาวน์โหลด Postmanเมื่อดาวน์โหลดแล้วให้ติดตั้งบนคอมพิวเตอร์ของคุณและเปิด Postman

Workspace และ Collection คืออะไร

Workspace เป็นแพลตฟอร์มการทำงานร่วมกันที่นักพัฒนาสามารถจัดระเบียบ จัดการ และทำงานร่วมกันในการทดสอบ Postman โดยให้พื้นที่ส่วนกลางในการสร้าง แก้ไข และดูแลรักษาทรัพยากร API

Collection ช่วยให้คุณสามารถจัดกลุ่มคำขอหลายรายการได้ ทำให้จัดการและดำเนินการสถานการณ์การทดสอบ Postman ได้ง่ายยิ่งขึ้น

วิธีการสร้างคำขอ? ในการเริ่มต้นใช้งาน Postman คุณสามารถสร้างคำขอ API ใหม่ภายใต้พื้นที่ทำงานที่ต้องการได้ เมื่อคุณตัดสินใจเลือกพื้นที่ทำงานแล้ว ให้เลือกประเภทคำขอ ตัวอย่างประเภทคำขอ ได้แก่ GET, POST, PUT, PATCH, DELETE เป็นต้น คำขอ GET ใช้เพื่อดึงข้อมูลจากเซิร์ฟเวอร์ API ในขณะที่ DELETE ใช้เพื่อลบข้อมูล คำขอ POST ใช้เพื่อแทรกข้อมูล ในขณะที่ PUT และ PATCH ใช้เพื่ออัปเดตข้อมูล

ขั้นตอนต่อไป ให้ป้อน URL Endpoint สำหรับการทดสอบ Postman. Endpoint คือจุดสื่อสารของ API และเป็น URL หรือ URI ภาพหน้าจอด้านล่างจะแสดงขั้นตอนการสร้างคำขอ GET
